# Road-Maker Sync

`tools/intersection-visualizer` is maintained as Cab87's Roblox-facing integration of
the upstream Road-Maker editor:

```sh
git remote add road-maker https://github.com/dbokser/Road-Maker.git
git fetch road-maker main
```

Use a review branch for every upstream sync:

```sh
git checkout main
git pull --ff-only
git checkout -b sync/road-maker-YYYYMMDD
git fetch road-maker main
git merge -s subtree -Xsubtree=tools/intersection-visualizer --allow-unrelated-histories --no-commit road-maker/main
```

Resolve conflicts in favor of Cab87-specific integration where needed:

- keep the `cab87-road-network` export schema, `version`, and Studio import settings;
- keep Roblox-facing settings such as `chamferAngleDeg` and `meshResolution`;
- keep local workflow fixes that do not exist upstream, such as merge-mode behavior;
- do not commit generated `dist`, `node_modules`, `.rbxl`, `.rbxlx`, or temporary Studio artifacts.

After resolving conflicts, run:

```sh
cd tools/intersection-visualizer
npm run lint
npm run build
cd ../..
rojo build default.project.json --output cab87.rbxlx
```

Review the diff before committing. Upstream Road-Maker is the source for editor mesh
math improvements; Cab87 remains the source for Roblox export/import and runtime bake
behavior.

## Mesher Parity Contract

The road graph mesher exists in two places:

- Road-Maker / intersection visualizer: `tools/intersection-visualizer/src/lib/meshing.ts`
- Roblox runtime/editor import: `src/shared/RoadGraphMesher.lua`

These implementations must stay behaviorally equivalent. Road-Maker is the authoring
reference for geometric mesh math. The Roblox module is a Luau port of that math plus
Roblox-facing data conversion. Runtime materialization belongs in
`src/shared/RoadMeshBuilder.lua`; it should not change the semantic mesh generated by
`RoadGraphMesher.lua`.

When changing meshing behavior, update both implementations in the same change set unless
the difference is explicitly Roblox-only materialization behavior. Do not fix a visual
artifact by adding a renderer-only workaround in one implementation while leaving the raw
mesh groups different.

## Mesh Data Groups

Both meshers should produce the same conceptual groups:

- `roadTriangles`: asphalt road surface, including edge road strips and junction hubs.
- `roadEdgeTriangles`: Roblox-side diagnostic/subgroup for non-hub road strips.
- `roadHubTriangles` / `hubTriangles`: junction road fan triangles.
- `sidewalkTriangles`: sidewalk surface.
- `crosswalkTriangles`: crosswalk surface.
- `roadPolygons`: road strip polygons and their inner/outer curves.
- `hubs`: junction records with `polygon`, `outerPolygon`, `corners`, and `outerCorners`.
- `crosswalks`: crosswalk polygons used for display/debug and fill boundary behavior.
- `polygonTriangles` / `polygonFills`: polygon-fill triangles and color metadata.
- `centerLines`: sampled route/navigation centerlines.

Names do not have to be identical when one side has compatibility aliases, but the
geometry represented by each group should match.

## Order Of Operations

Keep this order aligned with Road-Maker:

1. Sample edge splines.
2. Build node junction geometry: corners, outer corners, hub polygons, clearances, and
   squared edge bases.
3. Build road strips, sidewalks, crosswalks, road polygons, outer road curves, and
   centerlines.
4. Build polygon fills from road outer curves and hub outer polygons.
5. Compute bounds.

Polygon fills depend on data generated earlier in the pipeline. If the fill mesh overlaps
crosswalks or junction roads in Roblox but not in Road-Maker, first compare the boundary
inputs used by polygon fill stitching: `roadPolygons.*Curve`, `hubs.outerPolygon`, and
`crosswalks`.

Junction smoothing must also stay aligned. Road-Maker computes each corner smoothness as
`edge.transitionSmoothness + node.transitionSmoothness`; Roblox import/export must preserve
node `transitionSmoothness` and `RoadGraphMesher.lua` must use the same sum when calculating
junction corners and edge clearances.

## Hub Polygon Ordering And Winding

This area is easy to break.

- `hub.outerPolygon` must preserve Road-Maker construction order. Polygon-fill stitching
  walks this ordered ring to choose the correct path around junctions. Do not angle-sort
  `outerPolygon` just because it helps another output group.
- If Roblox needs a sorted hub ring for stable junction road fan triangulation, keep that
  sorted ring local to hub road triangle generation or store it separately. Do not feed it
  back into polygon-fill stitching.
- Triangle winding is a rendering concern, but it must still be deterministic. Roblox
  MeshParts may expose reversed triangles that are hard to see from above. If a junction
  fan renders from the underside, flip the triangle vertex order for that fan without
  changing the hub boundary order used by fills.
- Road-Maker's Three.js display often uses `DoubleSide`; do not use that as proof that
  winding is irrelevant for Roblox.

## Polygon Fill Rules

Polygon fills should match Road-Maker's generated fill mesh, not just its final layered
display.

- Fills are generated from authored fill node loops.
- For each fill edge that follows a road edge, the fill boundary uses the chosen
  `outerLeftCurve` or `outerRightCurve` from `roadPolygons`.
- Target-end edge bases are returned in target-node local order. Road-Maker reverses them
  before appending target points to road polygons and curves; Roblox must do the same or
  polygon fills can stitch across the junction road surface.
- At junctions, the fill boundary stitches between adjacent road curves by walking
  `hub.outerPolygon` in the selected direction.
- Crosswalk starts/ends modify road bases before `roadPolygons` are emitted, so a correct
  Road-Maker-parity fill boundary should not include crosswalk-owned surface area.
- Do not use centroid clipping, render offsets, or material priority as the first fix for
  polygon fill overlap. Those can hide duplicate geometry but do not preserve parity. Use
  them only for clearly documented Roblox-only materialization constraints.

## Renderer Versus Mesher

Do not infer mesh parity from normal rendered output alone.

Road-Maker has a renderer and debug layers on top of the raw mesh. Roblox turns mesh
groups into real runtime `MeshPart` instances. A surface that appears hidden in one
renderer may still exist in raw mesh data. When debugging parity, inspect the raw groups:

- In Road-Maker, hide/show layer debug views and inspect `mesh.polygonTriangles`,
  `mesh.crosswalkTriangles`, `mesh.roadTriangles`, and `mesh.hubs`.
- In Roblox, use the client road visual debug logs from
  `src/client/AuthoredRoadVisuals.client.lua` to compare triangle counts and generated
  MeshParts.
- Compare counts, bounds, and representative triangle coordinates before changing
  materialization.

## Validation Checklist

For any mesher change:

1. Make the same geometric change in `tools/intersection-visualizer/src/lib/meshing.ts`
   and `src/shared/RoadGraphMesher.lua`, unless the difference is documented as
   Roblox-only materialization.
2. Export a representative graph from Road-Maker with roads, junctions, smooth
   transitions, crosswalks, sidewalks, and polygon fills.
3. Import it into Studio and compare:
   - road triangle count
   - hub triangle count
   - sidewalk triangle count
   - crosswalk triangle count
   - polygon fill triangle count
   - mesh bounds
4. Check polygon fills with roads/crosswalks hidden in both tools to verify the raw fill
   geometry, not only layered rendering.
5. Run:

```sh
cd tools/intersection-visualizer
npm run lint
npm run build
cd ../..
rojo build default.project.json --output cab87.rbxlx
```

If `tools/intersection-visualizer` has unresolved sync conflicts, resolve those before
claiming parity. If a Roblox-only runtime issue remains after parity is confirmed, keep
that fix in `RoadMeshBuilder.lua`, the Studio plugin, or runtime scripts rather than in
the shared geometric mesher.
